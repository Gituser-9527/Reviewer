# Escalation Process

## 升级触发条件

- 高风险误判影响真实岗位发布。
- 发现未脱敏敏感信息进入日志、反馈或导出报告。
- 规则版本、知识库版本或模型版本记录缺失。
- 审核员与 Agent 结论大规模不一致。
- 客户或内部团队对依据准确性提出重大质疑。

## 分级

- P0：数据泄露、批量错误拦截、无法回滚。
- P1：严重误杀/漏判、证据错误影响多个 tenant。
- P2：单点流程问题、体验问题、低风险规则偏差。

## 处理流程

1. 记录 Beta feedback，标注严重度。
2. 通知 Beta owner 和合规负责人。
3. 必要时暂停 `limited_enforce`，回退到 `assist` 或 `shadow`。
4. 收集 auditRunId、ruleVersion、lawKbVersion、证据和人工复核结论。
5. 形成修复方案，并补充 eval/red-team 回归样本。
6. 修复后重新运行 build、test、eval 和 Go/No-Go 检查。

## 沟通原则

- 对外使用审慎表述：存在风险、建议复核、建议修改。
- 不输出“企业已经违法”等绝对法律结论。
- 对已知限制和临时降级措施保持透明。
